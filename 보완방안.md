## 1. Judge (최종 판정) 로직의 한계 및 개선 방안

**현재 상태 (약점)**
- `nodes/judge.py`에서 서술형 채점자 3명(엄격, 관대, 키워드)의 점수 중 **수학적 중앙값(Median)**을 최종 점수로 채택하고 있습니다.
- 중앙값은 극단적인 오작동(Outlier)을 걸러내는 데는 좋지만, **"채점자의 구체적인 감점 이유"를 무시하는 치명적인 단점**이 있습니다. 예를 들어 '엄격' 에이전트가 학생 답안의 치명적 개념 오류를 발견해 2점을 주었어도, 나머지 두 에이전트가 이를 놓치고 8점, 9점을 주면 그냥 8점으로 통과되어 버립니다.

**개선 방안 (Next Step)**
- **LLM 기반 합의(Consensus) 도출**: 단순 수식 계산을 폐기하고, 3명의 채점 결과(점수+근거)를 Judge AI에게 프롬프트로 전달하여 "세 명의 의견을 종합적으로 읽어보고 가장 타당한 비판을 채택해 최종 점수와 피드백을 작성하라"고 지시하는 방식으로 변경해야 합니다.
- **사용자 맞춤형 가중치 부여**: 학습자의 수준에 따라 초보자에게는 '관대함' 비중을, 숙련자에게는 '엄격함' 비중을 높여 계산하는 로직 추가가 가능합니다.

## 2. 비용(Cost) 및 응답 속도(Latency) 이슈

**현재 상태 (약점)**
- 문제 생성 시 4회(초안, 검증, 난이도, 판정), 채점 시 최소 4회(루브릭 생성 + 채점 3회)의 LLM API 호출이 발생합니다.
- 학생이 하나의 문제를 풀고 피드백을 받기까지 대기 시간이 길며, API 토큰 비용이 가파르게 증가합니다.

**개선 방안 (Next Step)**
- **문제 은행(DB) 캐싱 도입**: AI가 검증까지 마쳐 성공적으로 생성해 낸 문제는 폐기하지 않고 DB에 저장합니다. 이후 다른 학생이 동일한 개념의 문제를 요청하면, 재생성하지 않고 DB에서 즉시 꺼내어(Cache Hit) 비용과 속도를 극적으로 절감해야 합니다.
- **비동기(Async) 병렬 처리**: 서술형 채점 에이전트 3명의 API 호출을 현재처럼 직렬이 아닌 병렬(Async)로 호출하여 대기 시간을 3분의 1로 줄일 수 있습니다.

## 3. 문제 생성 AI의 한계점

**① 중복 문제 출제 가능성 (기억력 부재)**
- 현재 문제 생성 에이전트는 사용자가 직전에 풀었던 문제를 기억하지 못합니다. "다익스트라"를 반복해서 입력하면, 문장 구조만 조금 다른 동일한 질문을 반복해서 만들어낼 위험이 있습니다.
- **해결책**: 생성 프롬프트에 `session_history`(지금까지 푼 문제 목록)를 주입하여 "이전에 푼 문제와 겹치지 않는 새로운 시각의 문제를 출제하라"는 제약을 추가해야 합니다.

**② 코딩형/객관식 문제의 구조적 불안정성**
- 생성 AI가 "코딩형"을 만들 때, 완벽하게 동작하는 `test_cases` 배열(입력값/기댓값 쌍)을 JSON 포맷으로 뱉어낼 수 있을지 아직 검증이 부족합니다.
- 객관식 문제의 경우 정답을 제외한 매력적인 오답(Distractor)을 논리적으로 잘 구성하는지에 대한 프롬프트 세부 조정이 필요합니다.

**③ RAG 청크 단위의 한계 (문맥 단절)**
- 400단어 단위로 문서를 잘라 검색하므로, 단원 전체의 맥락을 종합해야 하는 "거시적이고 복합적인 서술형 문제"를 출제하기 어렵습니다. 주로 국지적이고 단편적인 지식 위주의 문제만 생성될 가능성이 큽니다.

## 5. 사용자 맞춤형 채점 모드 선택 기능 (향후 추가 가능)

**현재 상태**
- 서술형 채점 시 시스템이 강제로 3명의 에이전트(엄격, 관대, 키워드)를 모두 거치도록 고정(Hard-coded)되어 있습니다.

**개선 방안**
- 학습자가 문제를 풀기 전(혹은 시스템 설정에서) **"어떤 모드로 채점받을지"**를 직접 선택할 수 있는 기능을 제공합니다.
  - **관대한 연습 모드**: `grading_graph`의 라우터가 답안을 `critique_lenient` 에이전트에게만 보내어, 초보자가 의미만 맞추면 점수를 주고 자신감을 얻을 수 있게 합니다.
  - **깐깐한 실전 모드**: 시험이 임박한 사용자를 위해 답안을 `critique_strict` 에이전트에게만 보내어, 정확한 용어와 개념이 없으면 가차 없이 감점하도록 합니다.
- **장점**: 이미 각 모드별 채점 노드가 분리되어 구현되어 있으므로, `graphs/grading_graph.py`에 간단한 조건부 라우팅(Conditional Edge)만 추가하면 즉시 서비스화할 수 있는 매우 비용 효율적인 기능입니다.

## 6. 종합 모의고사(Mock Exam) 출제 및 100점 만점 평가 시스템 (개발 완료)

**현재 상태 (해결됨)**
- 사용자가 범위를 지정하면 백엔드의 `core/exam.py`를 통해 다중 문항을 비동기(Async) 병렬 처리로 빠르고 효율적으로 출제하는 **모의고사 모드**가 추가되었습니다.

**구현 완료된 주요 기능**
- **범위 기반 자동 다중 문항 생성**: Orchestrator AI가 지정된 범위 내에서 개념 중복을 방지하여 세부 주제를 쪼갠 뒤, ThreadPoolExecutor를 이용해 다수의 문제(예: 5~10문제)를 동시 출제합니다.
- **배점 자동 스케일링**: 문항 수에 맞게 각 에이전트가 배점을 자동 조절하여 총점을 **정확히 100점 만점** 스케일로 환산합니다.
- **단원 밸런스 및 중복 방지 오케스트레이션**: 전체 모의고사 세트를 조율하여 한 개념에 치우치지 않는 밸런스를 맞췄습니다.
- **종합 성적 리포트 생성**: 학생이 모든 문항의 답안을 제출하면 전체 일괄 채점을 수행하고, AI 튜터가 전체적인 취약점과 우수한 점을 분석하여 종합 성적표(피드백)를 제공합니다. 프론트엔드의 별도 탭에서 이 모든 과정을 시각적으로 편하게 이용할 수 있습니다.

## 7. 문제지 및 성적표 PDF/Word 문서 출력 기능 (향후 추가 로드맵)

**현재 상태**
- 문제 생성 및 채점 결과가 웹 화면에만 표시됩니다. 실물 시험지처럼 인쇄하거나, 포트폴리오·과제 제출 등을 위해 파일로 저장하는 기능이 없습니다.

**개선 방안**

### 방안 A — 프론트엔드 클라이언트 사이드 (권장, 서버 불필요)

| 출력 형식 | 사용 라이브러리 | 특징 |
|---|---|---|
| **PDF** | `jsPDF` + `jspdf-autotable` | 텍스트 기반 PDF 생성, 한글 폰트 임베드 가능 |
| **PDF (고품질)** | 브라우저 `window.print()` + Print CSS | 브라우저 인쇄 엔진 활용, 별도 라이브러리 불필요 |
| **Word (.docx)** | `docx` + `file-saver` | 완전한 .docx 포맷, MS Word에서 편집 가능 |

**구현 방식 (프론트엔드)**
```bash
# 필요 패키지 설치
npm install jspdf jspdf-autotable docx file-saver @types/file-saver
```

- **PDF 내보내기**: 문제 생성 직후와 채점 완료 후 각각 "PDF 저장" 버튼을 노출합니다. `jsPDF`로 문제 번호·유형·내용·채점 기준(루브릭)·점수·항목별 피드백을 표준 A4 레이아웃으로 생성합니다. 한글 폰트(`NanumGothic` 등)를 Base64로 임베드하여 깨짐을 방지합니다.
- **Word 내보내기**: `docx` 라이브러리로 제목·본문·표(루브릭, 채점 결과)·피드백 단락을 구조화한 `.docx` 파일을 생성합니다. `file-saver`의 `saveAs()` 함수로 브라우저 다운로드를 트리거합니다.

### 방안 B — 백엔드 서버 사이드

| 출력 형식 | 사용 라이브러리 | FastAPI 엔드포인트 예시 |
|---|---|---|
| **Word (.docx)** | `python-docx` | `POST /export/docx` → `.docx` 파일 스트리밍 반환 |
| **PDF** | `WeasyPrint` 또는 `ReportLab` | `POST /export/pdf` → PDF 바이트스트림 반환 |

```bash
# 필요 패키지 설치
pip install python-docx weasyprint
```

**구현 방식 (백엔드)**
- 프론트에서 문제 데이터와 채점 결과를 JSON으로 `/export/docx` 또는 `/export/pdf` 엔드포인트에 POST합니다.
- 서버에서 `python-docx`로 문서를 조립하고 `StreamingResponse`로 파일을 반환합니다.
- 클라이언트는 응답을 받아 브라우저 다운로드로 저장합니다.

**출력 문서에 포함될 내용**
- 📄 **문제지**: 문제 번호, 유형 배지, 문제 본문, 난이도, 답안 작성란(빈칸)
- 📊 **채점 결과지**: 최종 점수, 신뢰도, 항목별 점수 및 피드백, 오답 진단, 다음 추천 학습
- 📋 **모의고사 성적표**: 총점(100점 만점), 문항별 점수 바 차트, AI 튜터 종합 피드백

**장점**: 이미 프론트에 모든 데이터가 갖춰져 있기 때문에, 라이브러리 설치 후 "내보내기" 버튼 하나만 추가하면 되어 구현 비용이 매우 낮습니다.

## 8. OCR 기반 손글씨 답안지 업로드 자동 채점 (향후 추가 로드맵)

**현재 상태**
- 현재 시스템은 웹 텍스트 박스에 직접 타이핑한 디지털 답안만 채점 가능합니다. 실제 시험 환경에서는 학생이 종이에 손으로 쓴 답안지를 제출하는 경우가 대부분이며, 이를 디지털로 처리하는 파이프라인이 없습니다.

**개선 방안 — OCR 기반 답안지 자동 인식 및 채점**

### 전체 플로우

```
[학생이 종이에 답안 작성]
        ↓
[스마트폰 사진 촬영 or 스캔 업로드 (JPG/PNG/PDF)]
        ↓
[프론트엔드] → 파일 업로드 UI (drag & drop 지원)
        ↓
[백엔드 /upload_answer 엔드포인트]
        ↓
[이미지 전처리] → 기울기 보정, 노이즈 제거, 이진화 (OpenCV)
        ↓
[OCR 텍스트 추출] → GPT-4o Vision 또는 Tesseract OCR
        ↓
[추출된 텍스트 → 기존 grading_graph 채점 파이프라인 연결]
        ↓
[채점 결과 반환 → 프론트 렌더링]
```

### OCR 엔진 비교

| OCR 엔진 | 한글 정확도 | 손글씨 지원 | 구현 난이도 | 비용 |
|---|---|---|---|---|
| **GPT-4o Vision** (권장) | ★★★★★ | ★★★★★ | 낮음 | 유료 (토큰 과금) |
| **Google Cloud Vision API** | ★★★★☆ | ★★★★☆ | 낮음 | 유료 (월 1,000회 무료) |
| **Tesseract OCR** | ★★☆☆☆ | ★★☆☆☆ | 중간 | 무료 (오픈소스) |
| **Naver CLOVA OCR** | ★★★★★ | ★★★★☆ | 낮음 | 유료 (한국어 특화) |

> 한글 손글씨 인식은 **GPT-4o Vision**이 압도적으로 우수합니다. 이미 프로젝트에서 OpenAI API를 사용하고 있으므로 추가 연동 비용이 없습니다.

### 구현 방식

**프론트엔드 추가 사항**
- 기존 텍스트 입력 박스 옆에 **"파일 업로드로 답안 제출"** 탭을 추가합니다.
- Drag & Drop 또는 파일 선택기로 이미지(JPG, PNG) 또는 PDF를 업로드합니다.
- 업로드 후 미리보기 이미지를 표시하고, OCR 인식된 텍스트를 편집 가능한 텍스트 박스에 보여줘 학생이 오인식 구간을 직접 수정할 수 있게 합니다.

**백엔드 추가 사항**

```bash
# 필요 패키지 설치
pip install python-multipart opencv-python pillow
```

```python
# 신규 API 엔드포인트 예시 (api.py에 추가)
@app.post("/upload_answer")
async def upload_answer(file: UploadFile = File(...), problem_id: str = Form(...)):
    # 1. 이미지 수신 및 저장
    contents = await file.read()
    
    # 2. OpenCV 전처리 (기울기 보정, 이진화)
    img = preprocess_image(contents)
    
    # 3. GPT-4o Vision으로 OCR
    extracted_text = ocr_with_gpt4o_vision(img)
    
    # 4. 인식된 텍스트 반환 (사용자 확인 후 채점)
    return {"extracted_text": extracted_text, "preview_image": encode_base64(img)}
```

**GPT-4o Vision OCR 핵심 로직 (신규 노드: `nodes/ocr.py`)**

```python
def ocr_with_gpt4o_vision(image_bytes: bytes) -> str:
    """GPT-4o Vision으로 이미지에서 답안 텍스트 추출"""
    client = get_openai_client()
    base64_img = base64.b64encode(image_bytes).decode('utf-8')
    
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "user",
            "content": [
                {"type": "text", "text": "이 이미지에서 학생이 손으로 쓴 답안 텍스트를 정확하게 추출하세요. 줄바꿈과 구조를 최대한 유지하되, OCR 오류가 발생할 수 있는 부분에는 [?]를 표시하세요."},
                {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{base64_img}"}}
            ]
        }],
        max_tokens=1000
    )
    return response.choices[0].message.content
```

### 답안지 구조 인식 고도화 (선택적 확장)

단순 OCR을 넘어 **문항 구조 자동 인식**도 가능합니다.

- **모의고사 답안지 자동 분리**: "Q1. ..." "Q2. ..." 형식으로 답안지가 작성된 경우, GPT-4o Vision이 각 문항별 답안을 자동으로 분리하여 해당 문제와 1:1 매칭하여 채점합니다.
- **OMR 카드 지원**: 객관식 마킹 시트(OMR)를 업로드하면 OpenCV로 마킹된 원을 감지하여 선택 번호를 자동 인식합니다.
- **혼합 답안지 처리**: 한 페이지에 객관식(마킹)과 서술형(손글씨)이 혼합된 경우, 영역을 분리하여 각각 다른 방식으로 처리합니다.
